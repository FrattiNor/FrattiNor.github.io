我们都知道background属性是设置背景的属性，但是其实用到最多的也不过就是background-color，对我而言其他的属性用到很少。

background-image: 设置背景图像, 可以是真实的图片路径, 也可以是创建的渐变背景;
background-position: 设置背景图像的位置;
background-size: 设置背景图像的大小;
background-repeat: 指定背景图像的铺排方式;
background-attachment: 指定背景图像是滚动还是固定;
background-origin: 设置背景图像显示的原点[background-position相对定位的原点];

background-position
background-position属性用来设置背景图像的位置, 默认值: 0% 0%

设置两个值, 第一个值作用于横坐标, 第二个值作用于纵坐标, 取值可以是方位关键字[left\top\center\right\bottom]也可以是百分比或长度, 百分比和长度可为设置为负值.

background-origin
background-position 定位时参考的原点, 默认值 padding-box , 另外还有两个值: border-box 和 content-box。

content-box:：区域不算padding和border
padding-box：区域不算border
border-box：区域不算margin

background-clip
用于指定背景图像向外裁剪的区域, 默认值 border-box , 另外还有两个值: padding-box 和 content-box。（背景展示的地方）

content-box:：区域不算padding和border
padding-box：区域不算border
border-box：区域不算margin

-webkit-background-clip: text;的效果就是背景颜色被剪裁到文字上面，只有文字会显示背景颜色。这个属性可以做类似ktv里面歌词视图的效果。
必须加上-webkit-前缀，不然浏览器不能正常渲染。

background-repeat
background-repeat 属性定义了图像的平铺模式。默认为repeat

repeat 默认。背景图像将在垂直方向和水平方向重复。
repeat-x 背景图像将在水平方向重复。
repeat-y 背景图像将在垂直方向重复。
no-repeat 背景图像将仅显示一次。
inherit 从父元素继承 background-repeat 属性的设置。

background-attachment
background-attachment 属性设置背景图像是否固定或者随着页面的其余部分滚动。

scroll 默认值。背景图像会随着页面其余部分的滚动而移动。
fixed 当页面的其余部分滚动时，背景图像不会移动。
inherit 从父元素继承 background-attachment 属性的设置。

JavaScript函数式编程和函数柯里化

Posted on 2018-10-10

JavaScript函数式编程和函数柯里化

关于函数在实际开发中的应用，大体可以总结为函数声明、函数表达式、匿名函数、自执行函数。

1. 函数声明

JavaScript中，有两种声明方式，一个是使用var、let或者count的变量声明，另一个是使用function的函数声明。

变量对象的创建过程中，函数声明比变量声明具有更为优先的执行顺序，即我们常常提到的函数声明提前，也就是变量提升的一部分。因此我们在执行上下文中，无论在什么位置声明了函数，都可以在同一个执行上下文中直接使用该函数。

2. 函数表达式

当使用变量声明的方式来声明函数时，就是我们常常说的函数表达式。函数表达的提升方式与变量声明一致。

与函数声明不同，函数表达式使用了var、let或者count进行声明，那么在确认他是否可以正确使用的时候就必须依照let的规则进行判断，即变量声明。使用let进行变量声明，其实是进行了两步操作。

由于声明方式的不同，导致了函数声明与函数表达式在使用上的一些差异需要我们注意，除此之外，这两种形式的函数在使用上并无不同。

3. 匿名函数

匿名函数，顾名思义，就是指的没有被显示进行赋值操作的函数。它的使用场景，多作为一个参数传入另一个函数中。

fn的第一个参数传入了一个匿名函数。虽然该匿名函数没有显示的进行赋值操作，我们没有办法在外部执行上下文中引用到它，但是在fn函数内部，该匿名函数赋值给了变量bar，保存在了fn变量对象的arguments对象中，我们可以在函数内部使用该匿名函数。

4. 自执行函数

自执行函数是一种特殊的匿名函数。

在ES5中，没有块级作用域，因此常常使用函数自执行的方式来模仿块级作用域，这样就提供了一个独立的执行上下文，结合闭包，就为模块化提供了基础。

根据作用域链的单向访问，外面可能很容易知道在这个独立的模块中，外部执行环境是无法访问内部的任何变量与方法的，因此我们可以很容易的创建属于这个模块的私有变量与私有方法。
共有变量和共有方法可以通过闭包的方式创建。

函数自执行的方式另外还有其他几种写法，如!function(){}()，+function(){}()

但是ES6引入了let和count间接的创建了块级作用域，自执行函数的作用感觉不大了。

函数参数传递方式

基本数据类型复制，是直接值发生了复制，因此改变后，各自相互不影响。但是引用数据类型的复制，是保存在变量对象中的引用发生了复制，因此复制之后的这两个引用实际访问的实际是同一个堆内存中的值。当改变其中一个时，另外一个自然也被改变。

当值作为函数的参数传递进入函数内部时，也有同样的差异。函数的参数在进入函数后，实际是被保存在了函数的变量对象中，因此，这个时候相当于发生了一次复制。

到底是按值传递还是按引用传递？实际上跟变量复制没有区别。

但是还有一点要注意，复制引用对象的时候，对象a复制给了b，b修改了属性值时，a会改变。因为a，b的地址是一样的，但是当你试图覆盖对象时，b={}，b会拥有一个新的地址，这时a不会改变。

函数式编程

函数式编程的思维主要是建议我们将会多次出现的功能封装起来以备调用。

我们往往会不由自主的使用命令式编程的风格来完成我们想要干的事情。因为命令式编程更加的简单，直白。例如我们现在有一个数组，现在想要找出这个数组中的所有类型为number的子项。当我们使用命令式编程思维时，可能就会直接这样做。

当在另外的时刻，想要找出另外一个数组中所有的子项时，我们不得不把同样的逻辑再写一次。当出现次数变多时，我们的代码也变得更加糟糕且难以维护。

我们可以将这个方法封装起来为一个方法

如果未来需求变动，或者稍作修改，我们只需要对getNumbers方法进行调整就可以了。而且我们在使用时，只需要关心这个方法能做什么，而不用关心他具体是怎么实现的。这也是函数式编程思维与命令式不同的地方之一。

函数式编程具有以下几个特征

1. 函数与其他数据类型一样

函数与其他数据类型一样，处于平等地位，可以赋值给其他变量，也可以作为参数，传入另一个函数，或者作为别的函数的返回值。

2. 用表达式而不用语句

表达式是一个单纯的运算过程，总是有返回值

语句是执行某种操作，没有返回值。

函数式编程要求，只使用表达式，不使用语句。也就是说，每一步都是单纯的运算，而且都有返回值。函数式编程期望一个函数有输入，也有输出。因此良好的习惯应该每个函数都要有返回值。

3. 纯函数

纯函数的概念：相同的输入总会得到相同的输出，并且不会产生副作用的函数。

副作用，指的是函数内部与外部互动，产生运算以外的其他结果。（最典型的情况，就是修改全局变量的值）

函数式编程强调没有”副作用”，意味着函数要保持独立，所有功能就是返回一个新的值，没有其他行为，尤其是不得修改外部变量的值。

4. 闭包

函数内部的变量对外部环境来说是封闭的，闭包的作用就是将函数内部的变量对外可以访问

这样函数fn使用后的返回值可以访问函数fn内的变量a。在函数fn外面访问fn内的变量a。

函数柯里化的概念很简单：只传递给函数一部分参数来调用它，让它返回一个函数去处理剩下的参数。

柯里化函数的作用

函数柯里化允许你分隔复杂功能变成更小更容易分析的部分。这些小的逻辑单元显然是更容易理解，然后你的应用就会变成干净而整洁的组合，由一些小单元组成的组合。
作用上基本上是先传入复用部分，返回带传入其余参数的函数。

  1. 提高通用性。
    当一个通用性很高的函数需要传入多个参数时，经常发生这种情况，重复的参数多次传入，但是经过函数柯里化后，可以先把重复的参数先传入。把返回的函数多次使用就行了。
  2. 固定易变因素。
    提前把易变因素，传参固定下来，生成一个更明确的应用函数。（最典型的代表应用，是bind函数用以固定this这个易变对象。）
    bind 方法 将第一个参数设置为函数执行的上下文，其他参数依次传递给调用方法，并动态创建返回一个新的函数。

柯里化的性能

  • 存取arguments对象通常要比存取命名参数要慢一点.
  • 一些老版本的浏览器在arguments.length的实现上相当慢.
  • 创建大量嵌套作用域和闭包函数会带来花销，无论是在内存还是速度上.

在柯里化的实现中，柯里化虽然具有了更多的自由度，但同时柯里化通用式里调用了arguments对象，使用了递归与闭包，因此柯里化的自由度是以牺牲了一定的性能为代价换来的。所以只有在情况变得复杂时，才是柯里化大显身手的时候。

JavaScript对象和原型链（_proto_和prototype的区别）

JavaScript对象和原型链

对象的概念

对象被定义为“无序属性的集合，其属性可以包含基本值，对象或者函数”。

也就是说，在JavaScript中，对象无非就是由一些列无序的key-value对组成。其中value可以是基本值，对象或者函数。

创建对象的方法有2种

如果要创建很多的类似的对象时，频繁的写重复的代码很不舒服，这点可以通过工厂模式解决。

但是这样做我们没有办法识别对象实例的类型。
为了解决这个问题，javaScript自带构造函数可以解决，在java等语言中，是有class的，在class的基础上创建实例，也就是class - object，在js中没有class（ES6新增了class，但跟多的是object的语法糖），是object - object的，而构造函数就充当了class的作用。

构造函数

构造函数本身跟普通的函数没有什么区别，唯一的区别的是使用的途径不一样，构造函数使用new关键字创建实例。

这样就创建了一个Person实例。

new关键字让构造函数具有了与普通函数不同的许多特点，而new的过程中，执行了如下过程：

  1. 声明一个中间对象。
  2. 将该中间对象的原型指向构造函数的原型。
  3. 将构造函数的this，指向该中间对象。
  4. 返回该中间对象，即返回实例对象。

中间对象的原型指向构造函数的原型这一步，就要提到js中一个比较难以理解的部分了，原型链。

在函数中有一个prototype属性，这个属性指向函数的原型，只有在函数被当做构造函数使用时，才会用到这个属性。

原型

说到原型不得不说一下这2个属性，很容易混淆，我就经常搞错，_proto_和prototype。

《JavaScript权威指南》的一段描述：

Every JavaScript object has a second JavaScript object (or null ,but this is rare) associated with it. This second object is known as a prototype, and the first object inherits properties from the prototype.

翻译过来就是每个JS对象一定对应一个原型对象，并从原型对象继承属性和方法。

_proto_

在JS里，万物皆对象。方法（Function）是对象，方法的原型(Function.prototype)是对象。因此，它们都会具有对象共有的特点。即：对象具有属性_proto_，可称为隐式原型。

_proto_属性的值就是它所对应的原型对象

prototype

方法这个特殊的对象，除了和其他对象一样有上述_proto_属性之外，还有自己特有的属性——原型属性（prototype），这个属性是一个指针，指向一个对象，这个对象的用途就是包含所有实例共享的属性和方法。原型对象也有一个属性，叫做constructor，这个属性包含了一个指针，指回原构造函数。

不像每个对象都有_proto_属性来标识自己所继承的原型，只有函数才有prototype属性。

当你创建函数时，JS会为这个函数自动添加prototype属性，值是一个有 constructor 属性的对象，不是空对象。而一旦你把这个函数当作构造函数调用（即通过new关键字调用），那么JS就会帮你创建该构造函数的实例，实例继承构造函数prototype的所有属性和方法（实例通过设置自己的_proto_指向承构造函数的prototype来实现这种继承）。

原型链

前面说了_proto_和prototype，构造函数的prototype指向构造函数的原型，构造函数new出来的实例对象的_proto_也指向构造函数的原型，所以原型链是从构造函数开始往上一级一级的生成了原型链。

构造函数的prototype指向function对象，function对象的prototype指向object对象，这样就形成了一个原型链。当然prototype属性是可以改变的，你可以把构造函数指向一个已经存在的对象，但最后都会指向Object.prototype。

Object.prototype是原型链的顶端，所有对象从它继承了包括toString等等方法和属性。

CSS3 @media

@media可用于根据一个或多个基于设备类型、具体特点和环境的媒体查询来应用样式。

@media规则由一个或多个媒体查询组成，每个媒体查询包含一个可选的媒体类型和若干媒体特性。多个查询可通过使用逻辑运算符以多种方式组合起来，且对大小写敏感。

@media规则仅当媒体查询计算为真时，即，当指定的媒体类型与正在显示文档的设备的类型匹配并且所有媒体特征表达式计算为真时，才应用相应的样式。

媒体类型

  1. all: 所有媒体
  2. braille: 盲文触觉设备
  3. embossed: 盲文打印机
  4. print: 手持设备
  5. projection: 打印预览
  6. screen: 彩屏设备
  7. speech: ‘听觉’类似的媒体类型
  8. tty: 不适用像素的设备
  9. tv: 电视

Media所有参数

  1. width: 浏览器可视宽度。
  2. height: 浏览器可视高度。
  3. device-width: 设备屏幕的宽度。
  4. device-height: 设备屏幕的高度。
  5. orientation: 检测设备目前处于横向还是纵向状态。
  6. aspect-ratio: 检测浏览器可视宽度和高度的比例。
  7. device-aspect-ratio: 检测设备的宽度和高度的比例。
  8. color: 检测颜色的位数。
  9. color-index: 检查设备颜色索引表中的颜色，他的值不能是负数。
  10. monochrome: 检测单色楨缓冲区域中的每个像素的位数。
  11. resolution: 检测屏幕或打印机的分辨率。
  12. grid: 检测输出的设备是网格的还是位图设备。

关键字

  2. not: 用来排除某种制定的媒体类型
  3. only: 用来定某种特定的媒体类型

学习JavaScript的内存空间和事件循环

Posted on 2018-08-26

JavaScript具有垃圾回收机制，所以对于前端开发来说，内存空间不是一个经常被提起的概念。所以我对于内存空间的了解也是非常的模糊。

学习内存空间，需要对堆，栈和队列有基本的了解

栈

栈和堆，可以理解为存储数据的地方，声明一个变量并赋值，这个变量的key-value就存储在堆和栈中，js中没有严格意义上的区分堆内存和栈内存，所以可以简单的理解为变量都存储在推内存中，但是在js执行上下文的时候，我们需要栈数据结构的存取方式来帮助我们理解。

简单理解栈的存取方式可以用八个字来表示，先进后出，后进先出。
用模型表示，可以用一个杯子来表示，往杯子加一样粗细的冰柱，先放进去的总算被后放进去的压住，所以先放的需要最后才能取出。

堆

堆的数据结构可以用书架来表示，书架上的书可以随意的取出，所以在对内存中，数据的存储是无序的。顺序不影响我们存取。

队列

理解队列的数据结构，主要是帮助理解js中的事件循环。
队列的数据结构和栈内存的数据结构相反，先进先出，后进后出，可以理解为一根水管，是有2个口的，一个入口和一个出口，而杯子只有一个出口，出口和入口在一起。

事件循环

js最大的特点是单线程的。也就是说，js的程序执行是线性的，不存在并发的情况，所以这个线程中只有位移的一个事件循环。

js的代码执行过程中，除了函数调用栈来搞定函数的执行顺序外，还依靠队列结构搞定其他代码。

任务队列分为宏进程 ( macro-task ) 和微进程 ( micro-task )。
宏进程大概包括：script（整体代码），setTimeout，setInterval，setImmediate，I/O，UI render。
微进程大概包括：process.nextTick，Promise，MutationObserver

所以，事件循环从开始执行代码到最后结束代码，可以简单的理解为，js有2条队列。一条宏队列，一条微队列，开始的时候，宏队列只有一个进程。微进程是空的，当宏进程执行到setTimeout等往宏队列里添加一个宏进程，执行遇到Promise.then等往微进程里添加一个微进程。最后当前宏进程执行结束后，进入微队列。把微队列里的微进程全部执行完后，在回到宏队列，执行下一个宏进程，以此循环，到代码执行结束为止。

需要注意的是，当执行微进程时，遇到微进程或宏进程时。跟在宏进程遇到它们一样，执行微进程时，遇到宏进程，依旧把宏进程丢进宏队列后面，但是遇到微进程时，也是会丢到微进程后面，但是会在这次微队列执行时一起执行掉，而不是到下一次的微进程。
